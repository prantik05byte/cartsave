# CartSave — Decision Log
**Track 2: AI-Assisted Checkout Recovery | Kasparro Agentic Commerce Hackathon**

A running record of every significant decision made during the build, in the format: considered X, chose Y, because Z.

---

| # | Decision | Options Considered | Choice Made | Reasoning |
|---|---|---|---|---|
| 1 | When to trigger the recovery widget | (A) Exit intent only (B) Time-based timer (C) Behavioral friction classifier | C — Behavioral friction classifier | Exit intent triggers too late (buyer already leaving). Timer triggers regardless of real friction (false positives). Behavioral signals map to specific blockers, enabling targeted opening messages. |
| 2 | Conversation format | (A) Multi-step survey form (B) Open-ended chatbot (C) Guided conversation with quick replies | C — Guided conversation | A survey feels like friction, not help. An open-ended bot gives buyers too much surface to go off-topic. Guided conversation balances naturalness with focus on resolving the specific blocker. |
| 3 | When to fetch Shopify cart data | (A) On session start (page load) (B) On friction trigger | A — Session start | Fetching at trigger adds 400–800ms of latency at the worst moment. Pre-fetching at load adds ~200ms to a non-critical moment and ensures the AI has context ready instantly when intervention starts. |
| 4 | Who executes Shopify API mutations | (A) Claude decides and calls API directly (B) Claude recommends action, backend executes it | B — Claude recommends, backend executes | Separates AI reasoning from side effects. Prevents Claude from making API calls with wrong parameters. Every action is validated and logged. AI/deterministic boundary is auditable. |
| 5 | Conversation length cap | (A) No cap (B) 5 turns (C) 8 turns (D) 12 turns | C — 8 turns | 5 turns is too short for complex issues (e.g., size + shipping confusion together). 12 turns would increase Claude API costs significantly. 8 turns covers 95% of resolvable cases in our test scenarios. After 8 turns without resolution, escalation to human support is the right outcome. |
| 6 | Widget default state | (A) Open by default, prominent (B) Minimized dot, expands on trigger | B — Minimized, expands on trigger | An open widget on every checkout would feel like an intrusive popup. Buyers would dismiss it reflexively. A minimized widget that activates specifically when friction is detected preserves the sense that the tool is responding to them, not broadcasting at them. |
| 7 | Database for session storage | (A) In-memory only (B) SQLite (C) PostgreSQL | B — SQLite for dev, PostgreSQL-compatible schema | In-memory does not survive server restarts. PostgreSQL requires Docker/cloud setup, slowing down hackathon dev. SQLite + Sequelize ORM gives real persistence with zero setup and a production-ready schema. |
| 8 | Handling Claude API downtime | (A) Show error message (B) Fallback to static responses by friction type | B — Friction-type fallback responses | An error message ("Our AI is unavailable") destroys buyer trust at the worst moment. A friction-specific static response is less good than AI but still useful and trustworthy. |
| 9 | Language support | (A) English + Hindi (two primary Indian e-commerce languages) (B) English only | B — English only | Multilingual prompting and UI localization would add 2–3 days of work. The hackathon demo is judged on product thinking and core functionality, not internationalization. Documented as a known limitation. |
| 10 | Analytics scope | (A) No analytics (B) API endpoint only (C) Full merchant dashboard UI | B — API endpoint only | Full UI is 1–2 days of additional work with no impact on the core track evaluation. The API endpoint proves the product concept and demonstrates that recovery rate is measurable. Merchant dashboard is documented as "next step" in limitations. |
| 11 | AI model choice | (A) OpenAI GPT-4o (B) Anthropic Claude Sonnet (C) Google Gemini | B — Claude Sonnet | Claude follows structured system prompts with high consistency, handles JSON output reliably, and has lower hallucination rate on grounded tasks in our testing. Also: Kasparro's own infrastructure context suggested comfort with Anthropic tooling. |
| 12 | Cart state on page refresh | (A) Restart session (B) Resume from sessionStorage | B — Resume if < 30 minutes | Restarting would lose conversation context and frustrate a buyer who was mid-conversation. sessionStorage persists across refreshes within the same tab. 30-minute TTL ensures stale sessions do not confuse returning buyers. |
